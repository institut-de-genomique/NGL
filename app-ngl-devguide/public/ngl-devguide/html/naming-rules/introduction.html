Introduction

Important avant lecture :

La documentation suivante présente les règles de nommage pour le développement de NGL et pour toutes les applications web de l'équipe développement.

Les règles suivantes sont le résultat de l'analyse des différents problèmes de nommage rencontrés au cours des développements de NGL. L'objectifs de ses règles est d'avoir une cohérence et une cohésion entre les différentes parties d'une application web.

Le code d'une application web appartient aux développeurs. Il est donc de la responsabilité de celui çi de respecter les règles de codage mais c'est surtout un devoir, en tant que développeur, d'améliorer continuellement le code.

Lors de vos développements, vous allez rencontrer des bouts de code ne respectant pas les règles définies. Il peut y avoir deux raisons valables à cela :

  1. Le code date d'avant les règles de nommage et n'a pas été remanié
  2. Les règles ont évolué et tout le code n'a pas été reprit

Pour ces bouts de code, il est du devoir du développeur de les remettre dans le droit chemin sans attendre. Il est donc vivement conseiller de faire le refactoring dès que l'intrus est démasqué et de ne pas remettre cela à plus tard même si le code n'est pas le votre.

De même pour cette documentation, elle doit vivre donc n'hésitez pas :

  • à mettre en évidence les incohérences.
  • à la corriger.
  • à améliorer les règles.
  • à ajouter des règles.

Les règles de nommage interviennent à tous les niveaux d'une application :

  • Package (controllers, view, etc.)
  • Nom des classes Controllers
  • Nom des classes View
  • Nom des méthodes des Controllers
  • Nom des répertoires pour les assets
  • Nommage des routes
  • Nommage des clés pour l'internationalisation (messages)
  • etc.

1. La ressource primaire

Pour organiser son code et appliquer la bonne nomenclature, il faut définir la ressource primaire sur laquelle le développement va se faire. La ressource est à prendre au sens REST du terme.

Pour cela, avant de commencer, il faut répondre à l'une des questions suivantes en fonction du développement à faire :

  • Qu'est ce que je dois afficher ?
  • Qu'est ce que je dois rechercher ?
  • Qu'est ce que je dois créer ?

La réponse à l'une de ses questions définie la ressource primaire.

Par exemples :

  • je souhaite afficher des tubes => la ressource est tubes.
  • je souhaite rechercher des processus => la ressource est processus.
  • je souhaite créer une liste de courses => la ressource est listes de courses .
  • etc.

Une fois la ressource primaire trouvée, il faut la mettre au pluriel et en anglais tout de suite. Dans les chapitres suivants nous utiliserons cette syntaxe pour y faire référence : <ResourceNames>

Remarque : Il est important de prendre 5 minutes pour faire cette réflexion car c'est beaucoup plus long de faire du refactoring et de se battre avec SVN ;-).
Remarque : Mettre les termes en anglais est obligatoire car pour définir des noms de variable c'est plus simple qu'en français et que de toute manière un language informatique est à base d'anglais.

2. L'organisation générale

Dans un programme en java qui utilise des packages pour organiser les classes, il y a deux possibilités pour organiser son code :

  • Soit en utilisant une organisation orientée par les parties techniques.
  • Soit en utilisant une organisation orientée par les parties métiers.

Les deux ont des avantages et des inconvenients qui ne seront pas listés ici. Ce qu'il faut retenir c'est que les 2 organisations se mélangent avec en fonction du contexte une dominante technique ou une dominante fonctionnelle.

Nous utilisons le framework Play qui s'appuie sur le design pattern MVC et ceci fait que nous avons une séparation technique des classes qui sont des controlleurs, des vues et des modèles dans 3 packages différents :

  • controllers
  • views
  • models

Au sein de ces 3 packages nous auront une organisation fonctionnelle par ressource primaire. Ceci permettra de regrouper dans un même sous package ce qui est lié à une ressource primaire. Cette façons de faire vous permet en tant que développeur de vous concentrer sur le métier. L'organisation de ses 3 packages sera abordée dans les chapitres suivants.

Nous aurons également une organisation plutôt orientée technique pour tout ce qui est commun à plusieurs métiers, comme des classes utilitaires ou des services transverses qui seront tous dans un même package.

3. Les exemples

Dans les chapitres suivants, les exemples s'appuirons sur une application qui doit permettre de réserver un hôtel.

Cette application doit permettre de :

  • Rechercher des hôtels et afficher les détails d'un hôtel.
  • Pouvoir afficher l'ensemble des chambres d'un hotel et les administrer (ajouter, supprimer, editer)
  • Faire une réservation d'hotel, rechercher des réservations et afficher les détails d'une réservation.
